HashMap的实现原理

发表于 2018-01-08 | 分类于 数据结构

1 什么是HashMap

HashMap是一种用于映射（键值对）的数据结构，底层实现是数组和链表，在jdk1.8之后，引入了红黑数和对扩容的优化机制。

2 简介

HashMap继承自抽象类AbstractMap，并且实现了Map接口，HashMap只能有一个k记录ey为null，允许多个记录值为null，HashMap并不是线程安全的，在多线程的情况下可以使用并发包下的ConcurrentHashMap。

3 内部实现

HashMap是一种key-value的数据结构，底层结构是数组+链表，在jdk1.8版本之后，链表的元素个数大于8时会转化为红黑树结构。

3.2HashMap几个重要的字段

  1. int threshold // 顾名思义：阈值,所能容纳的key-value对的极限。
  2. final float loadFactor //负载因子，默认的初始负载因子的大小是0.75,Node[]数组的默认长度是16，threhold = length*loadFactor。 当hashMap的实际容量大于threhold会发生resize操作,resize之后的容量是之前的两倍。默认的初始负载因子0.75是对时间效率和空间的一个平衡，如果在空间充足的情况下，对存取时间要求较高，可以调低loadFactor的值来减少hash碰撞，反之，可以加大loadFactor的值，并且这个值可以大于1。
  3. modCount 主要用来记录HashMap内部结构发生变化的次数，但是某个key对应的value值被覆盖不属于结构变化。

3.3HashMap设计

  • hashMap数组的长度设计为2的n次方（合数），这是一种非常规的设计，常规的设计是把桶的长度设计成素数来减少冲突。HashMap采用这种非常规设计，主要是为了在取模和扩容时做优化，同时为了减少冲突，HashMap定位哈希桶索引位置时，也加入了高位参与运算的过程。
  • 当hashMap链表过长时会严重影响hashMap的性能，jdk1.8版本之后，引入了红黑数的概念，当链表长度过长，默认超过8时，链表就转换为红黑树，利用红黑树快速增删改查的特点提高HashMap的性能。

    3.4方法实现

1.确定哈希桶数组的索引位置

对于增加，查找，删除，确定哈希桶的位置是第一步，也是关键的一步。理论上，当元素分布均匀，每个桶的位置上只有一个元素时，就无需遍历链表，提高性能。HashMap定位数组索引位置，直接决定了hash方法的离散性能。下面是源码的实现：

public V put(K key, V value) {
return putVal(hash(key), key, value, false, true);
}

final Node<K,V> getNode(int hash, Object key) {
    Node<K,V>[] tab; Node<K,V> first, e; int n; K k;
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (first = tab[(n - 1) & hash]) != null) {
        //第一个直接命中
        if (first.hash == hash && // always check first node
            ((k = first.key) == key || (key != null && key.equals(k))))
            return first;
        if ((e = first.next) != null) {
            if (first instanceof TreeNode)
            //是红黑树
                return ((TreeNode<K,V>)first).getTreeNode(hash, key);
            //遍历链表    
            do {
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    return e;
            } while ((e = e.next) != null);
        }
    }
    return null;
}

单例模式属于创建型模式，顾名思义，就是说一个类只有一个实例。

懒汉式（非线程安全）

上述代码使用了懒加载模式，但是在多线程的情况下会产生多个实例，存在并发问题。

懒汉式（线程安全）

上述方法，将类方法getInstance设置为同步，虽然是线程安全的，但是效率较低，每次调用该方法都会加锁，但是实际上我们只需要在第一次创建对象时加锁。

懒汉式(双重校验锁)

上述代码看似解决了第二种方法效率较低的问题，但实际上在多线程下还是存在问题。问题在于 new Instance() 并非是原子操作，在jvm中大概做了三件事：

  1. 给instance分配内存。
  2. 调用Singleton的构造函数，初始化成员变量。
  3. 将instance对象指向分配的内存空间（这一过程之后对象非null）
    但是JVM在编译过程中会进行指令重排序，上面的2和3的顺序是不确定的。如果顺序是1-3-2，在2未执行前，被另一个线程抢占，这是instance已经是非null了（但是没有初始化完成），所以线程二会直接返回instance。

volatile有两个作用:

  1. 保证变量的可见性，每次获取变量的值时都会从主存中去刷新。
  2. 禁止指令的重排序。
    所以对于一个volatile变量的写操作都先行发生于后面对这个变量的读操作。(java5中屏蔽重排序才得以修复)。

饿汉式

上述写法单例会在加载类后一开始就被初始化，但是这个写法存在缺陷，比如说我们在得到实例之前无法调用某个方法给他设置参数。

这种写法使用了jvm自身机制来实现线程安全，由于SingletonHolder是私有的，只能通过getInstance()方法来访问，读取实例的时候不会进行同步，不存在性能问题。

设计模式之适配器模式

发表于 2017-08-09 | 分类于 设计模式

1、定义

将一个类的借口转化成客户期望的另一个接口,适配器让原本借口不兼容的类可以相互合作。

2、涉及的角色

  • 目标角色：期待得到的接口。
  • 源角色：需要适配的接口。
  • 适配器角色：把源接口转化成目标接口。

3、例子

以充电器为实例，手机充电器一般需要5v左右的电压，家用交流电220v，所以需要一个适配器。

3.1、一部手机

public class Mobile{
  public void inputPower(V5Power v5Power){
    int provideV5Power = v5Power.provideV5Power();
  }
}

3.2、手机提供了一个5v电压的接口：

public interface V5Power｛
  public int provideV5Power();
｝

3.3、家用交流电

public class V220Power｛
  public int provideV220Power(){
    return 220;
  }
｝

3.4、适配器

public class V5PowerAdaper implements V5Power{
  private V220Power v220Power;
  public V5PowerAdaper(V220Power v220Power){
    this.v220Power = v220Power;
  }
  public int provideV5Power(){
    int power = v220Power.provideV220Power();
    ...
    ...
    return 5;
  }
}

3.5、给手机充电

public class Test  
{  
    public static void main(String[] args)  
    {  
        Mobile mobile = new Mobile();  
        V5Power v5Power = new V5PowerAdapter(new V220Power()) ;   
        mobile.inputPower(v5Power);  
    }  
}

3.6、优点

  1. 将目标类和适配着类解耦
  2. 增加了类的透明性和复用性，将具体的实现封装在适配着类中，对于客户端来说是透明的，提高了适配着的复用性。
  3. 灵活性和扩展性都非常好，符合开闭原则。

设计模式之观察者模式

发表于 2017-08-08 | 分类于 设计模式

1.模式定义

定义了对象之间一对多的依赖,当一个对象改变时,它的所有依赖着都会受到通知并且更新

2.模式结构

观察者模式包含的角色:

  • 被观察者:可以是接口,可以是抽象类或者具体的类
  • 观察者:一般是一个接口,在观察者状态发生变化时,这个方法就会被触发调用
  • 具体的被观察者:使用这个角色为了便于扩展,可以在角色中定义具体的业务逻辑
  • 具体的观察者,观察者接口的具体实现,定义了被观察者对象状态发生变化时索要处理的逻辑

3.被观察者

abstract class Subject{
    private Vector obs = new Vector();
    public void addObserver(Observer obs){
        this.obs.add(obs);
    }
    public void deleteObserver(Observer obs){
        this.obs.remove(obs);
    }
    protected void notifyObserver(){
        for( Observer o : obs){
            o.update;
        }
    }
    public abstract void doSomething();
}

4.具体的被观察者

class ConcreteSubject extends Subject{
    public void doSomething(){
        this.notifyObserver();
    }
}

5.观察者

interface Observer{
    public void update();
}

6.具体的观察者

class ContreteObserver1 implements Observer{
    public void update(){
        System.out.println("观察者1被通知");
    }
}
class ContreteObserver2 implements Observer{
    public void update(){
        System.out.println("观察者2被通知");
    }
}

7.调用

public class Client{
    public static void main(String[] args){
        Subject subject =new ConcreteSubject();
        subject.addObserver(new ContreteObserver1());
        subject.addObserver(new ContreteObserver2());
        subject.doSomething();
    }
}

8.观察者模式的优点

  • 观察者和被观察者之间是属于轻度的关联关系,并且是抽象耦合的,容易扩展
  • 常用的触发机制,依次对各个观察者进行处理,同时也是观察者模式的一个缺点,属于链式触发,当观察者比较多时性能不佳

设计模式之策略模式

发表于 2017-07-29 | 分类于 设计模式

1. 定义

策略模式属于对象的行为模式。针对一组算法，将每一个算法封装到具有共同接口的独立的类中，从而使得它们可以相互替换。策略模式使得算法可以在不影响到客户端的情况下发生变化。

2. 策略模式的结构：

策略模式涉及到三个角色：

  • 环境角色:持有对一个Strategy的引用
  • 抽象策略角色:这是一个抽象角色，通常由接口或抽象类实现
  • 具体策略角色:包装了相关的算法或者行为

3. 源代码

环境角色类:

public class Context {
  //持有一个具体策略的对象
  private Strategy strategy;
  //构造函数，传入一个具体策略对象
  //@param strategy    具体策略对象
  public Context(Strategy strategy) {
    this.strategy = strategy;
  }
  //策略方法
  public void contextInterface() {
    strategy.strategyInterface();
  }
}

抽象策略类：

public interface Strategy {
  // 策略方法
  public void strategyInterface();
}

具体策略类：

public class ConcreteStrategyA implements Strategy {
   @Override
  public void strategyInterface() {
       //相关的业务
      }
  }
  public class ConcreteStrategyB implements Strategy {
     @Override
      public void strategyInterface() {
       //相关的业务
     }
  }

4. 认识策略模式：

策略模式的优点：

  1. 策略模式提供了管理相关的算法族的办法。策略类的等级结构定义了一个算法或者行为族恰当使用继承可以把公共的代码移到父类里面，从而避免代码重复。
  2. 可以避免使用多重条件语句。

策略模式的缺点：

  1. 客户端必须知道所有的策略类，并自行决定使用哪一个策略类。这就意味着客户端必须理解这些算法的区别，以便适时选择恰当的算法类。换言之，策略模式只适用于客户端知道算法或行为的情况。
  2. 由于策略模式把每个具体的策略实现都单独封装成为类，如果备选的策略很多的话，那么对象的数目就会很可观。
